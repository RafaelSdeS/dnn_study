# CNN Kernel Research — Coding Instructions

## Core Principles

1. **Be concise and focused** — Address only what's asked; avoid over-engineering
2. **Write clear, maintainable code** — Prefer readability over cleverness
3. **Make minimal necessary changes** — Don't refactor, add features, or change architecture unless explicitly requested
4. **Preserve existing style** — Match indentation, naming, structure, and patterns in the current codebase
5. **No unsolicited documentation** — Don't write docs, tutorials, or migration guides unless asked
6. **Prefer simple solutions** — Use straightforward approaches; don't introduce complexity
7. **Explain significant changes** — If a major refactor or architectural change is needed, explain why and ask first

## Project Conventions

### Notebooks & Experiments
- **Separate save directories**: `saved_models/`, `saved_models_2/`, `saved_models_small_kernels/`, `saved_models_hybrid_qat/`
- **Reusable training**: Use `ml_utils.train_epoch()` and `ml_utils.evaluate()` for consistency
- **Configuration**: Use `ExperimentConfig` from ml_utils for all hyperparameters
- **Seed management**: Always call `set_seed(42)` at notebook start for reproducibility

### Code Style
- Include docstrings with clear Args/Returns for public functions (see `ml_utils.py` pattern)
- Use type hints where applicable
- Organize code into logical sections with clear comments
- Use `torch.cuda.empty_cache()` between long-running experiments
- Prefer `torch.amp.autocast()` for mixed precision training

### Model Definitions
- Inherit from `nn.Module` and implement `forward()`
- Use consistent naming: `SmallKernelCNN`, `Kernel2x2CNN`, etc. (verb + descriptor + Layer type)
- Add residual blocks for deeper small-kernel networks to mitigate gradient issues
- Document kernel size rationale in class docstring if non-standard

### Data Loading
- Use `ImageNet normalization`: `mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225]`
- Apply augmentation only to training split
- Batch size: typically 64 (adjust for OOM)
- Use `num_workers=4, pin_memory=True, persistent_workers=True` for efficiency

### Metrics & Analysis
- Use `count_parameters(model)` to report parameter efficiency
- Compare models on: accuracy, parameter count, and training time
- Save results to JSON in experiment directory for later aggregation
- Track both FP32 baseline and INT8 quantized accuracies

## When to Ask or Explain

Before proceeding, ask or explain when:
- A requested change requires architectural refactoring
- Modifying shared code (`ml_utils.py`) that affects multiple notebooks
- Introducing new dependencies
- Changing existing save paths or directory structure
- Removing existing functionality
